- **Core concept:** Build a *custom* organizational SSO / Identity Provider (IdP) — own frontend + backend — speaking OIDC. Explicit decision NOT to deploy an off-the-shelf IdP (Keycloak / Authentik). Org context: EnvOcc (environmental & occupational health). [Rationale for build-vs-buy: TBD — being explored.]
- **Stakes:** Production / real org. Real EnvOcc users & systems depend on it → held to production-grade security rigor.
- **Scale:** ~100–150 internal staff users (single org).
- **Build-vs-buy rationale (RESOLVED):** Mix of (3) org mandate that code be in-house [primary, load-bearing justification], (4) team wants to own/understand the auth layer, and (2) full control over login UX/branding/flows. Off-the-shelf (Keycloak/Authentik) ruled out. → Brief MUST include an explicit "risk accepted + mitigation" section, since a mandate manages custom-IdP security risk rather than removing it.
- **Build-vs-buy DECISION FINAL (do not relitigate):** Coach noted Keycloak supports custom frontends via themes (so driver #2 alone wouldn't require a custom build) and probed how hard the #3 mandate is. User chose to proceed regardless. Decision is settled: **build the custom in-house OIDC IdP.** Working mode remains Coaching path. Mandate (#3) taken as given going forward.
- **Identity store:** NOTHING exists today → the IdP becomes the **system of record**. It will store and manage credentials itself → heaviest security burden (password hashing, reset, lockout, MFA all in-house). [Implication for brief risk section.]
- **Existing apps:** ~10–15 internal apps, **each with its own separate login** today. Consolidating these behind one SSO is the core pain. Implies an identity-reconciliation + user-migration story (same ~100–150 people likely hold multiple scattered accounts). App stacks: TBD.
- **IdP build stack:** Deferred to the architecture phase (brief = WHAT, not HOW). Svelte tooling present in workspace but stack NOT decided. Backend/DB: TBD.
- **Identity reconciliation (A=Yes):** Largely the SAME ~100–150 people hold accounts across multiple of the 10–15 apps. SSO must collapse duplicate per-app accounts into ONE canonical identity per person → dedup/reconciliation is a real v1 concern.
- **Provisioning (B=Admins):** Admin-driven account creation. Admins create the ~100–150 accounts and assign roles. No password migration possible (apps hash differently) → implies an invite / first-login password-set (activation) flow.
- **Lifecycle owner (C=HR Admins):** HR Admins are the source of truth for "who is a valid employee" and own joiner/mover/leaver (onboarding new staff, disabling leavers). → Brief implies an **admin console** for HR admins (user CRUD, role assignment, enable/disable) as a first-class part of the build, not just the login pages.
- Session set to **max effort** (user ran /effort).
- **Scope levers decided (v1):**
  - **Authz scope = AUTH ONLY.** IdP proves identity & issues OIDC tokens; each app keeps managing its own roles/permissions internally. Role/group centralization explicitly OUT of v1 (candidate for vision/phase 2). Keeps scope tractable.
  - **MFA = YES in v1** (e.g. TOTP authenticator). Required given production + sole identity store.
  - **Password reset = SELF-SERVICE** (email link). → hard dependency on reliable org email/SMTP.
  - **Rollout = PHASED / pilot first.** Wire 1–2 apps, prove in production, then migrate the rest of the 10–15.

  - **App integration ownership = APP TEAMS/OWNERS.** This project delivers the IdP + integration docs/guide; each app owner wires OIDC into their app. Clean boundary keeps scope smaller. → "OIDC integration guide for app owners" is a v1 deliverable.
  - **Email/SMTP = reliable, available.** Self-service password reset and first-login activation are unblocked.
