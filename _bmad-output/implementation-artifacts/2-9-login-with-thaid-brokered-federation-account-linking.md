# Story 2.9: Login with ThaiD (brokered federation & account linking)

Status: ready-for-dev

## Story

As a staff member,
I want to sign in with ThaiD instead of a password,
so that I can use my national digital ID and skip the password + code.

**Epic:** 2 — Staff Authentication & SSO Identity
**GH Issue:** #15

**Scope boundary:** This story adds a second, alternative authentication path — "Login with ThaiD" — via native Keycloak identity brokering (an external OIDC Identity Provider configured in `realm-export.json`), alongside the existing email+password+TOTP path (Stories 2.2/2.6, untouched by this story). It does **NOT** integrate the real DOPA/ThaiD service — DOPA RP onboarding and the exact claims ThaiD asserts (including the PID claim name/format) are an open external unknown (PRD OQ1/OQ3/OQ4, architecture.md Decision 2 residual R8a), confirmed out of scope here. Instead, dev/CI/this story's own tests run against a **mock OIDC IdP** container added to `compose.yaml`. It does **NOT** build the HR Admin "capture PID at account creation" UI (Story 4.4, Epic 4, not yet built) — this story's own integration tests pre-register the federated-identity PID link directly via Keycloak Admin REST calls (the same mechanism Story 4.4 will call from its UI later), so this story is self-contained and does not block on unbuilt Epic 4 work. It does **NOT** write any custom Keycloak SPI code (Java/JS) — the entire brokering + linking mechanism is realm-level config-as-code (an `identityProviders` entry + a custom, declaratively-assembled `authenticationFlows` entry built from Keycloak's built-in flow/execution building blocks), consistent with NFR8 ("no hand-rolled auth logic") and architecture.md Decision 2 ("no custom auth code").

## Acceptance Criteria

**AC1 — "Login with ThaiD" button shown as a secondary option (FR13a, UX-DR6)**
Given the Sign-in surface renders,
When the `envocc` realm has the `thaid` identity provider configured and enabled,
Then a "Login with ThaiD" button is shown below the email/password form, visually subordinate (secondary-styled) to the primary sign-in action, preceded by a divider/heading separating it from the form (DESIGN.md `thaid-login-button` component spec: full-width within the auth-card, `rounded.md`, keyboard-focusable with the standard focus ring).

**AC2 — Keycloak brokers the flow top-level; Keycloak remains the IdP (FR13a, FR2)**
Given ThaiD is configured as an external OIDC Identity Provider in realm config-as-code (`keycloak/realm-export.json`, `identityProviders[]`, alias `thaid`, `providerId: "oidc"`),
When a user chooses "Login with ThaiD,"
Then the browser is redirected top-level (never framed/embedded, per FR2/FR12) to the configured `authorizationUrl`, the user authenticates at the external IdP (ThaiD in production, the mock OIDC IdP in dev/CI), and on successful return Keycloak — not the external IdP — issues its own ID/access/refresh tokens bearing the same `sub`/claim contract as the password path (Story 2.3). No credentials or tokens from ThaiD are ever exposed to or usable by the relying-party application.

**AC3 — First-time ThaiD login links to the pre-created account by PID; both paths resolve to the same identity (FR13a, FR21)**
Given a staff account was pre-created (Story 4.4, out of this story's scope, but simulated by this story's own test fixtures) with a federated-identity link pre-registered via `POST /admin/realms/envocc/users/{id}/federated-identity/thaid` (`{"identityProvider": "thaid", "userId": "<PID>", "userName": "<PID>"}`),
When that same PID authenticates via ThaiD (or the mock IdP configured to assert that PID as its subject claim) for the first time,
Then Keycloak's broker matches the existing federated-identity link and signs the user directly into the **same** pre-created local account — `sub` in the resulting token equals that account's Keycloak user `id`, identical to what a password-path login for that same account would return. A second ThaiD login by the same PID reuses the same link with no re-prompt and the same `sub`. A ThaiD identity whose PID has **no** pre-registered federated-identity link does **not** create a new ("phantom") account — the custom first-broker-login flow denies it cleanly (see Dev Notes — PID-Linking Design).

**AC4 — Disabled account blocked via ThaiD exactly as via password (FR25)**
Given an account is disabled (`enabled: false`),
When it attempts to authenticate via "Login with ThaiD" (using a PID with a valid pre-registered federated-identity link to that account),
Then brokered login is blocked — Keycloak rejects the authentication before any token is issued — identically to how Story 2.8 proved `enabled: false` blocks the ROPC/password path. This story adds the brokered-path proof (Story 2.8 only tested ROPC/browser password grants, not the broker path).

**AC5 — Dev/CI exercises the full brokered path via a mock OIDC IdP, no DOPA dependency (AR8 / architecture.md Decision 2)**
Given dev/CI has no access to the real DOPA/ThaiD service,
When the ThaiD flow runs in this environment,
Then the `thaid` IdP's `authorizationUrl`/`tokenUrl`/`userInfoUrl`/`issuer`/`jwksUrl` point at a mock OIDC IdP container added to `compose.yaml` in this story, configurable per test user to assert an arbitrary PID as its subject claim — exercising AC2/AC3/AC4 end-to-end without any real ThaiD dependency. The mock IdP responds to standard OIDC discovery (`/.well-known/openid-configuration`).

## Dev Notes

### PID-Linking Design — the central decision in this story

Read this before touching `realm-export.json`. The naive reading of AC3 ("a first-broker-login flow links it to the pre-created staff account by national ID (PID)") could be misimplemented as "use Keycloak's default first-broker-login flow, which links by matching a user attribute or email." **That is wrong for this realm** — `keycloak/IDENTITY-MODEL.md` (Story 2.1) is explicit that PID must never appear in Keycloak user `attributes` (it lives only in the identity-provider federated-link subsystem), so no attribute-matching or email-matching mechanism can find "the pre-created account by PID."

The actual mechanism (confirmed against `_bmad-output/test-artifacts/test-design/test-design-epic-2.md` R-007 mitigation and its Assumption #4 — "ThaiD PID is stored as a Keycloak identity provider link attribute, not as a top-level user attribute"):

1. **Pre-registration (Story 4.4's future job, simulated by this story's own test fixtures):** when HR creates a staff account with a captured PID, they (or, for now, this story's tests) call `POST /admin/realms/envocc/users/{id}/federated-identity/thaid` with `{"identityProvider": "thaid", "userId": "<PID>", "userName": "<PID>"}`. This creates a `FederatedIdentity` row keyed on `(identityProvider, providerUserId)` pointing at the local user — this IS "PID stored only in the broker link, not in attributes."
2. **Real ThaiD login, PID already pre-registered:** when the external IdP returns an identity whose subject/PID claim matches an existing `FederatedIdentity.userId`, Keycloak's **core broker logic** (not the named "First Broker Login" flow) finds the match and logs straight into that account. Keycloak's technically-named "First Broker Login" flow **only executes when no such link exists yet** for that (provider, providerUserId) pair.
3. **Real ThaiD login, PID NOT pre-registered:** this is when the custom `thaid-first-broker-login` flow (Task 2) actually runs. Its job is to guarantee this case does **not** silently create a new local account (a "phantom account" — explicitly named as an exploratory risk in the test-design doc) and does not fall back to email-based auto-linking (which would be wrong here — this realm's linking key is PID, never email, for the ThaiD path).

So "first-broker-login flow links it to the pre-created staff account by national ID" (AC3's wording) is best read as: **the overall PID-linking mechanism (steps 1-2) is what performs the linking; the named "first-broker-login flow" (step 3) is the deny-only safety net for the unmatched case.** Do not build a flow that tries to actively perform PID matching — Keycloak's core broker engine already does that via the pre-registered federated-identity link; the flow only needs to fail closed when that link doesn't exist.

### Mock OIDC IdP — chosen approach and its limits

No mock IdP exists in this repo at this story's baseline (`compose.yaml` has exactly three services: `postgres`, `keycloak`, `nginx`). The test-design doc flagged this as a dependency that should have been resolved "before Story 2.9 work begins" (owner: Rawinan) but it was not — so choosing and wiring it is folded into this story's Task 0. `navikt/mock-oauth2-server` is recommended: MIT-licensed, actively maintained, purpose-built for exactly this (mocking OAuth2/OIDC providers in integration tests), and supports per-request custom claims — needed because TS-290b/d/e each require asserting a *different* PID/subject value from the mock IdP within the same test run. **Confirm the exact current tag + pin by `@sha256:` digest at implementation time** (this story specifies the image family, not a blind pin — consistent with how architecture.md's Decision 1 versions were "web-verified... re-pin at finalize").
