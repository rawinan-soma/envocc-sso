## Part 2: Challenges and Growth Areas

### Key Struggles

**1. Story 1.5 had the most post-implementation churn (7 fix commits after initial feat commit).**

Amelia (Developer): "Story 1.5 had the most turbulence. After the initial feat commit, there were 7 fix commits: SAST permissions, CI branch trigger corrections, realm-lint harden, test helper gap, threshold parity, script path args. That's a story where the full-system integration surface — local git hooks + GitHub Actions + Python scripts + BATS tests — created more edge cases than anticipated."

Alice (Product Owner): "Some of that was the CI environment being a different runtime than local. GitHub Actions has permission models that don't surface until you push."

Amelia (Developer): "Exactly. The `contents: read` permission for the SAST job — that only fails in CI, not locally. We don't have a good way to catch that in the pre-commit hook."

**2. The missing `compose_service_field` helper caused 4 tests to silently fail.**

Murat (TEA Agent): "The test review caught a critical defect: `compose_service_field()` was called via `declare -f` in 4 tests (TS-138a/b/c/d) but never defined in `common.bash` at the time. Tests appeared syntactically valid but silently failed at runtime. This was fixed in a follow-up commit (`9df3703`), but it highlights a gap: test helper completeness wasn't validated before the initial story implementation."

Amelia (Developer): "Good catch. The root cause is that the test review happened after implementation, not as part of the green-phase verification. We should make helper completeness a checklist item in dev-story before marking a story ready for review."

**3. Story 1.2 had a gitleaks false positive that needed investigation.**

Amelia (Developer): "The `.env` not-git-ignored finding was a false positive — `git check-ignore` confirmed `.env` was correctly ignored. But it consumed investigation time. The `.gitleaks.toml` allowlist patterns were correct all along; the reviewer noted it without running the check first. Small waste, but worth noting."

**4. CI branch triggers didn't match the actual Git Flow until deferred.**

Amelia (Developer): "The CI triggers used `push: [main, develop, story-*]` but the actual branch model is `main/dev/epic-N`. A PR from `epic-1 → dev` would not match the `pull_request: [main, develop]` trigger. This was flagged in Story 1.2's deferred work and carried into Story 1.5 for reconciliation. Story 1.5 fixed the branch names."

Alice (Product Owner): "The fix landed in the right place, but it took two stories to get there. Something to address earlier for Epic 2."

## Part 3: Key Insights

**Insight 1: Keycloak 26 runtime vs. build-time configuration is a recurring gotcha.**
The `KC_HEALTH_ENABLED` build-time requirement was the most critical technical discovery of the epic. It was documented thoroughly in Story 1.1 and prevented re-encounters, but Epic 2 stories will configure KC heavily — verify which settings are build-time vs. runtime in KC 26 (Quarkus) before each story that modifies the Dockerfile or KC configuration.

**Insight 2: CI-only behaviors can't be caught locally.**
Some defects (SAST `contents:read` permission, `hashFiles()` at job-level if vs. step-level if) are only discoverable in the actual CI environment. For Epic 2 stories that expand CI gates, consider running `act` locally for dry-run simulation before pushing.

**Insight 3: The code review process is the quality gate, not the initial implementation.**
Stories 1.1, 1.2, 1.3, and 1.5 all had meaningful patches from code review. The initial implementation is a draft; the review is where correctness is established. This is working as designed — the key is ensuring reviews are thorough and patches are applied before the story is marked done.

**Insight 4: Clean reference material = clean delivery.**
Story 1.4 (design tokens) had zero debug churn because the DESIGN.md frontmatter was the precise, single source of truth for every token value. Compare to stories where spec was implicit or distributed across multiple documents.

**Insight 5: Deferred work accumulates and needs tracking.**
`deferred-work.md` has 6 items from Stories 1.1 and 1.2. Several are relevant to Epic 2 (Postgres maintenance DB isolation, `pg_isready` assertion strength, integration test volume isolation). These should be reviewed during Epic 2 story creation to decide which get promoted to stories.

## Part 5: Epic 2 Preview & Dependencies

### Significant Dependencies to Verify

The `--import-realm` with `IGNORE_EXISTING` strategy means `docker compose down -v` is required to re-apply changes. Confirm the team is using this discipline.

2. **`deferred-work.md` items D1 (Postgres maintenance DB isolation) and D4 (integration test volume isolation)** — assess whether these are blocking for Epic 2 integration tests.

3. **CI branch triggers are now reconciled** (fixed in Story 1.5) — verify the `epic-2` branch pattern is covered before starting Epic 2.

### Epic 2 Change Detection

No architectural assumptions from Epic 1 were proven wrong during implementation. Epic 2's current plan is sound. One watchpoint: Epic 2 Story 2.5 (branded login theme) adds files to `keycloak/` — ensure the Dockerfile `COPY` path for design tokens is verified at that story's dev notes time, not discovered mid-implementation.
