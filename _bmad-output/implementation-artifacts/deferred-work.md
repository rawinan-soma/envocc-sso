## Deferred from: code review of story-1.1 (2026-06-25)

- **Healthcheck `cat <&3` has no client-side read timeout** [compose.yaml] — The Keycloak readiness probe reads the management socket until EOF, relying on the server honoring `Connection: close`. Already bounded by Docker `healthcheck.timeout: 10s` and confirmed by live boot, so low risk. Consider wrapping the read in `timeout`/`read -t` during infra hardening.
- **Maintenance DBs (`postgres`, `template1`) remain PUBLIC-connectable by both roles** [postgres/init/01-init-databases.sh] — AC2's pairwise isolation (keycloak ↔ admin) is fully enforced. For stricter least-privilege, add `REVOKE CONNECT ON DATABASE postgres, template1 FROM PUBLIC` (and re-grant the superuser as needed). Defer to Story 1.5 / infra hardening.
- **`pg_isready` healthcheck does not assert the `keycloak` DB exists** [compose.yaml] — Postgres can report healthy before init scripts finish in theory; mitigated in practice because the official entrypoint does not open the TCP listener until init completes. A stricter gate (`pg_isready -d keycloak` or a `SELECT`) would make the AC1 ordering guarantee explicit.
- **Integration tests share the developer's named volume / no isolated compose project** [tests/] — `compose_down_volumes` in the test helpers operates on the same `postgres_data` volume as a developer's running stack. Give the test stack its own compose project name / volume namespace when the CI gate lands (Story 1.5).

## Deferred from: code review of story-2.5 (2026-06-27)

- **AC2 literal "exactly one CSP header" — Keycloak default CSP passes through alongside nginx's** [keycloak/realm-export.json, nginx/nginx.conf] — The realm sets no `browserSecurityHeaders`, so Keycloak applies its built-in default `contentSecurityPolicy` (`frame-src 'self'; frame-ancestors 'self'; object-src 'none';`), and the nginx auth blocks (`/realms`, `/auth`, `/admin`) `add_header` their own CSP without `proxy_hide_header Content-Security-Policy`. Net result: two CSP headers on auth pages, the Keycloak one carrying a conflicting `frame-ancestors 'self'`. Framing is still blocked (browsers intersect both CSPs, and `X-Frame-Options: DENY` is set), so this is a contract/cleanliness gap, not a security hole. Pre-existing from epic-1 (realm defaults + nginx story 1.3); not introduced by story 2.5 (which only added `loginTheme`). Fix during infra hardening: add `proxy_hide_header Content-Security-Policy;` to the three nginx auth location blocks (story-1.3 domain).
- **Credential error not displayed on the `usernameHidden` re-auth path** [keycloak/themes/envocc/login/login.ftl] — When `usernameHidden??` is true and the password is wrong, the global feedback area is suppressed (`displayMessage=!messagesPerField.existsError('username','password')`) and the only inline error span lives inside the non-hidden username branch, so no error renders. Mirrors upstream Keycloak base `login.ftl`; only affects the reauthentication flow, which the email+password sign-in path does not exercise. Add a password-group error span if a reauth surface is introduced.
